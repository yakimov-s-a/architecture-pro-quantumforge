# Задание 1. Исследование моделей и инфраструктуры

Компания работает в Европе, поэтому попадает под требования GDPR. Передача персональных данных, технических
спецификаций и других документов должна удовлетворять требованиям законов. Американские провайдеры моделей (к примеру
OpenAI) обязаны предоставлять любые данные, которые проходят через их сервисы, по запросу властей. Поэтому передача
данных чувствительных данных из Европы через таких провайдеров недопустима.

Соответственно, необходимо выбирать провайдеров, которые соответствуют требованиям GDPR, или разворачивать модели
локально.

Ограничения на выбор провайдера определяет доступный набор моделей.

## 1. Сравнение LLM-моделей

### Модели OpenAI

Одна из наиболее зрелых экосистем и одни из наиболее развитых моделей. На данный момент лучшая предоставляемая модель —
GPT-5.5. Стоимость от \$5/$30 за 1 млн. токенов ввода/вывода.

Для бота с RAG хорошо подойдет более "слабая" модель. К примеру GPT-4.1 Mini — она имеет контекстное окно в 1 млн.
токенов (как и GPT-5.5), но стоит существенно дешевле (\$0.4/$1.6 ввод/вывод)

**Плюсы:**

* одна из самых развитых экосистем инструментов;
* простое взаимодействие через API;
* нет требований к собственной инфраструктуре.

**Минусы:**

* данные уходят на серверы OpenAI в США;
* стоимость даже дешевой модели может быть значительной при активном использовании.

**Применимость:** нельзя использовать для бота, который оперирует чувствительной информацией.

### YandexGPT

Модель YandexGPT 5.1 Pro имеет контекстное окно в 1 млн токенов. Стоимость — около \$6.5/$6.5 за 1 млн. токенов
ввода/вывода. По результатам бенчмарков, модель сравнима с GPT-4.1.

**Плюсы:**

* работа с русскоязычными текстами, хорошая поддержка кириллицы;
* хорошая интеграция с Yandex Cloud.

**Минусы:**

* данные обрабатываются на российских серверах — это юридический риск для финско-эстонской компании с европейскими
  клиентами;
* качество поддержки английского языка уступает OpenAI.

**Применимость:** нельзя использовать для бота, который оперирует чувствительной информацией.

### Локальные модели (Hugging Face)

Стоит начать эксперименты с моделями малого размера (4-40 миллиардов параметров) и перейти к средним (40-150 миллиардов
параметров) в случае недостаточного качества результатов. Стоимость развертывания и поддержки моделей большего
размера (>150 миллиардов параметров) будет неоправданной для нашего решения.

Среди моделей малого размера на данный момент лучшие результаты в бенчмарках показывают модели Qwen 3.6 (от 27 до 35
миллиардов параметров) и Gemma 4 (от 26 до 31 миллиарда параметров).

Кандидаты:

| Модель      | Параметры | Бенчмарки |
|-------------|-----------|-----------|
| Qwen3.6 27B | 27B       | Хорошо    |
| Qwen3.6 35B | 35B       | Отлично   |
| Gemma 4 26B | 26B       | Хорошо    |
| Gemma 4 31B | 31B       | Хорошо    |

Все четыре модели распространяются под лицензией Apache License 2.0.

**Плюсы:**

* данные не покидают периметр;
* нет зависимости от изменения стоимости внешнего провайдера;
* возможность производить fine-tuning;
* контроль версий и квантования моделей.

**Минусы:**

* модели среднего и большого размера требуют дорогих серверов с большим количеством GPU-памяти;
* добавляется сложность в развертывании и поддержке локальных моделей.

**Применимость:** лучшее решение для компании.

## 2. Сравнение моделей эмбеддингов

### Облачные: OpenAI Embeddings

| Модель                 | Размерность | Цена (1 млн. токенов) | Бенчмарк (MTEB) |
|------------------------|-------------|-----------------------|-----------------|
| text-embedding-3-small | 512–1536    | $0.02                 | Хорошее         |
| text-embedding-3-large | 256–3072    | $0.13                 | Отличное        |

**Плюсы:**

* простое взаимодействие через API;
* нет требований к собственной инфраструктуре.

**Минусы:**

* проблемы с конфиденциальностью при работе из Европы.

**Применимость:** не подходит, так как при использовании OpenAI Embeddings весь индексируемый текст отправляется на
серверы OpenAI, — это проблема конфиденциальности, такая же как и с LLM.

### Локальные: Sentence Transformers

Sentence Transformers — это библиотека со множеством доступных предобученных моделей.

Так как компания работает с несколькими Европейскими странами, нам нужно поддерживать различные языки. Модель
`paraphrase-multilingual-mpnet-base-v2` будет хорошим выбором в данной ситуации. Она имеет размерность 768 и достаточно
небольшой размер, что позволяет запускать ее локально без существенных затрат на инфраструктуру.

**Плюсы:**

* мультиязычность;
* достаточно высокое качество результатов;
* независимость от внешних провайдеров API.

**Минусы:**

* индексация базы существующих документов потребует некоторого времени.

**Применимость:** лучшее решение для компании.

## 3. Сравнение векторных баз данных

### FAISS

FAISS — это библиотека для быстрого поиска сходства по векторам, не является полноценной базой данных.

**Плюсы:**

- очень высокая скорость поиска (все данные находятся в памяти);
- полная автономность;
- высокая конфигурируемость;

**Минусы:**

* для очень больших объемов индексов требуется слишком много памяти;
* нет метаданных и механизмов работы с ними;
* необходимо самостоятельно реализовывать механизмы синхронизации данных между разными машинами.

**Применимость:** хорошо подходит для старта экспериментов, но не подходит для конечного продукта из-за отсутствия
метаданных.

### ChromaDB

ChromaDB — это полноценная векторная база данных, созданная специально для RAG-приложений.

Для текущего количества документов и страниц, количество хранимых векторов будет достаточно небольшим.

Оценим пессимистично (в реальности, объем будет меньше);

1. у нас около 20000 документов;
2. пусть каждый из них содержит около 30 страниц;
3. пусть каждая страница будет делиться примерно на 5 чанков;
4. тогда количество векторов можно рассчитать как 20000 * 30 * 5 = 3000000;
5. при точности F32 и векторах из 768 компонентов, каждый вектор будет занимать 3072 байта;
6. общий размер хранилища будет равен 3000000 * 3072 = 9216000000 байт или 8.5 гигабайт.

**Плюсы:**

* хранит векторы и метаданные в одном месте;
* хорошо подходит для RAG с фильтрацией по различной информации;
* встроенная персистентность без дополнительного кода;
* готовые решения для деплоя в облако.

**Минусы:**

* медленнее FAISS (хуже для экспериментов, но все еще очень быстрая для реальной работы).

**Применимость:** подходит в качестве основного хранилища векторов.

## 4. Конфигурация сервера

### Вариант 1. Полностью внешние провайдеры (OpenAI API + ChromaDB на EKS)

| Компонент    | Решение                | CPU        | RAM    | GPU |
|--------------|------------------------|------------|--------|-----|
| LLM          | GPT-4.1 Mini           | -          | -      | -   |
| Эмбеддинги   | text-embedding-3-small | -          | -      | -   |
| Векторная БД | ChromaDB               | 4 vCPU AWS | 16 GiB | -   |

**Плюсы:**

* быстрый старт;
* минимальная операционная нагрузка.

**Минусы:**

* все документы уходят в OpenAI — недопустимо для чувствительных данных.

**Применимость:** неприемлемо из-за отправки данных в OpenAI.

### Вариант 2. Полностью локально с небольшой моделью — Qwen3.6-35B A3B Q4_K_M + paraphrase-multilingual-mpnet-base-v2

| Компонент    | Решение                               | CPU         | RAM    | GPU         |
|--------------|---------------------------------------|-------------|--------|-------------|
| LLM          | Qwen3.6-35B A3B Q4_K_M                | 16 vCPU AWS | 32 GiB | NVIDIA H100 |
| Эмбеддинги   | paraphrase-multilingual-mpnet-base-v2 | 4 vCPU AWS  | 16 GiB | -           |
| Векторная БД | ChromaDB                              | 4 vCPU AWS  | 16 GiB | -           |

**Плюсы:**

* данные хранятся внутри существующего контура компании;
* невысокая и полностью предсказуемая стоимость решения.

**Минусы:**

* квантованная модель может давать недостаточно качественные результаты.

**Применимость:** удовлетворяет всем требованиям.

### Вариант 3. Полностью локально с небольшой моделью — Qwen3.6-35B + paraphrase-multilingual-mpnet-base-v2

| Компонент    | Решение                               | CPU         | RAM    | GPU             |
|--------------|---------------------------------------|-------------|--------|-----------------|
| LLM          | Qwen3.6-35B                           | 16 vCPU AWS | 32 GiB | NVIDIA H200 x 2 |
| Эмбеддинги   | paraphrase-multilingual-mpnet-base-v2 | 4 vCPU AWS  | 16 GiB | -               |
| Векторная БД | ChromaDB                              | 4 vCPU AWS  | 16 GiB | -               |

**Плюсы:**

* данные хранятся внутри существующего контура компании.

**Минусы:**

* достаточно высокая стоимость из-за высоких требований к объему памяти GPU.

**Применимость:** вариант с хорошим качеством ответов и высокой стоимостью.

### Лучший вариант

Предпочтительно выбрать вариант 2 (локальные модели с квантованием). Он позволит запустить решение с минимальными
затратами. При необходимости улучшить качество ответов, в будущем можно будет перейти на более крупную LLM без
существенных изменений в архитектуре решения.
